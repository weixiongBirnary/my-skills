# 分层锚点导航文档生成

为项目生成三层锚点文档，以最小 token 消耗最大化 AI 对项目的理解与定位能力。

## 输入要求

必须：项目目录树（≥2 层深度）
推荐：技术栈说明
可选：模块职责描述、已有 README

如果用户未提供目录树，先用工具扫描项目结构再开始生成。

## 输出结构

按顺序生成 3 层文档。若项目较小（<5 模块），可合并第一、二层为单文件。

---

### 第一层：`PROJECT_MAP.md`（项目级）

**约束：≤300 字，禁止大段叙述**

格式：

```markdown
# PROJECT_MAP

> {一句话项目定位} / {技术栈} / {架构风格}

## 模块索引

| 模块 | 路径 | 入口 | 职责（≤20字） | 依赖 |
|------|------|------|---------------|------|
| Auth | `src/auth/` | `index.ts` | JWT登录+RBAC权限 | UserModule |

## 全局约定
- 类型定义：`src/types/`
- 公共工具：`src/common/`
- 配置入口：`src/config/index.ts`
```

### 第二层：模块级 README（每个核心模块）

**约束：每个 ≤200 字**

格式：

```markdown
# {模块名}

> {1-2句职责描述}

## 文件职责
- `index.ts` → 公共 API 导出
- `xxx.service.ts` → {职责}

## 对外接口
- `Service.method(params): ReturnType`

## 内部流程
{用箭头链表示：Request → Guard → Service → Repository}
```

### 第三层：文件级头注释模板

**约束：≤10 行/文件**

格式：

```
/**
 * @file {filename}
 * @desc {一句话职责}
 * @exports {对外暴露的符号}
 * @boundary {不应在此文件做的事}
 * @depends {直接依赖}
 */
```

## 生成规则

1. **信息密度优先**：同等信息量选 token 更少的表达
2. **路径必须准确**：基于用户提供的目录树或工具扫描结果，不猜测
3. **职责边界清晰**：每个模块同时说明"做什么"和"不做什么"
4. **禁止冗余**：不写"该模块是系统重要组成部分"类无效描述
5. **可增量更新**：格式支持手动追加，不依赖特定工具
6. **中文输出**：除代码和路径外使用中文

## 超出范围

- 不生成业务逻辑代码
- 不做代码质量分析或重构建议
- 不生成 API 文档（Swagger 等）
- 不替代 CHANGELOG

## 工作流

1. 获取/扫描项目目录树
2. 识别核心模块和入口文件
3. 生成 PROJECT_MAP.md
4. 逐模块生成 README
5. 为入口文件和核心文件生成头注释模板
6. 末尾说明哪些文件建议加入 `.cursorrules` 常驻引用

## 适配策略

| 项目规模 | 策略 |
|----------|------|
| 小型（<5 模块） | 合并第一、二层为单文件 |
| 中型（5-15 模块） | 标准三层 |
| 大型（>15 模块） | 按域/子系统分组，每组一个 PROJECT_MAP 章节 |

## 与 project-anchor-map 的关系

| 维度 | project-anchor-map | layered-anchor-docs（本 skill） |
|------|-------------------|-------------------------------|
| 侧重 | 代码流分析、入口锚点挖掘 | 分层导航文档生成 |
| 输出 | 调用路径、依赖、风险 | 三层结构化文档 |
| 场景 | 理解"系统怎么跑的" | 帮 AI 快速定位"什么在哪" |

两者可配合使用：先用本 skill 建立导航骨架，再用 project-anchor-map 深入分析关键流程。
