# 互联网工作团队工作流程

> 适用团队：产品 × 1、组长 × 1、开发 × 1、设计 × 1、测试 × 1

---

## 一、整体流程阶段

| 阶段 | 负责人 | 核心产出 |
|------|--------|----------|
| 📋 需求分析 | 产品主导，组长统筹（全员参与） | PRD 需求文档、排期计划 |
| 🎨 设计阶段 | 设计主导（全员评审） | UI 稿、标注稿、切图资源 |
| 💻 开发阶段 | 开发主导 | 技术方案、可测试包 |
| 🧪 测试阶段 | 测试主导，产品验收（开发配合） | 测试报告、Bug 闭环、验收结论 |
| 🚢 上线发布 | 组长 + 开发执行，产品收尾 | 线上验证、数据分析、迭代规划 |

---

## 二、各阶段工作流

### 📋 阶段一：需求分析
1. 产品进行市场调研，收集用户需求，撰写 **PRD 需求文档**
2. 组长召开需求评审会（全员参与），对齐目标与范围
3. 评审未通过 → 产品修改 PRD；通过 → 组长输出**排期计划**与任务拆解分配

### 🎨 阶段二：设计阶段
1. 设计根据 PRD 产出**原型图 / UI 设计稿**
2. 组织设计评审（产品 + 组长 + 开发 + 测试参与）
3. 评审未通过 → 返回修改；通过 → 输出**标注稿与切图资源**

### 💻 阶段三：开发阶段
1. 开发完成**技术方案设计**
2. 前后端**并行开发**，提升整体效率
3. 开发完成后进行**自测 & 联调**
4. 达到提测标准后，输出测试包并正式**提测**

### 🧪 阶段四：测试阶段
1. 测试提前（需求阶段起）编写**测试用例**
2. 执行功能测试、兼容性测试、性能测试
3. 发现 Bug → 提交 Bug 报告 → 开发修复 → 回归验证（循环）
4. 测试通过 → 进入**产品验收环节**
5. 产品验收不通过 → 提出优化意见 → 开发修复 → 重新验收
6. 产品验收通过 → 输出**测试报告**

### 🚢 阶段五：上线发布
1. 组长审批上线申请，确认上线条件已满足
2. 开发执行**部署上线**，监控服务运行状态
3. 全员进行**线上验证**，确认功能符合预期
4. 产品收集**用户反馈与数据埋点分析**，形成数据闭环
5. 组长 + 产品联合主导**项目复盘**，输出下一轮**需求迭代规划**

---

## 三、协作关键原则

### 1. 🔀 并行推进
- 设计出稿后，开发的技术方案设计与设计走查可**同步进行**，缩短整体周期

### 2. 🔍 测试左移
- 测试人员从**需求阶段**就介入，提前识别风险，减少后期返工成本

### 3. 🎯 产品验收前置
- 产品在测试通过后、上线前进行验收，确保功能符合用户预期，**避免上线后大规模返工**

### 4. 📅 每日同步
- 建议每日站会（≤ 15 分钟），同步各成员**进度与阻塞点**，快速响应问题

### 5. 📁 文档沉淀
- PRD、设计稿、测试用例、上线记录、数据报告全部**归档留存**，保障信息可追溯

### 6. ✅ 明确 DoD（完成定义）
- 每个阶段设定明确的**完成标准**，避免理解偏差与责任扯皮

---

## 四、关键流转节点

```
PRD 评审通过 → 设计评审通过 → 开发提测 → 测试通过 → 产品验收通过 → 上线审批 → 复盘归档 → 迭代规划
```

> 每个节点均需相关负责人**确认签收**，确保交接无遗漏。

## 五、角色协作矩阵

| 阶段 | 产品 | 组长 | 开发 | 设计 | 测试 |
|------|------|------|------|------|------|
| 需求分析 | 主导 | 统筹 | 参与 | 参与 | 参与 |
| 设计阶段 | 评审 | 评审 | 评审 | 主导 | 评审 |
| 开发阶段 | 答疑 | 跟进 | 主导 | 走查 | 准备 |
| 测试阶段 | 验收 | 跟进 | 修复 | 走查 | 主导 |
| 上线发布 | 分析 | 主导 | 执行 | 验收 | 回归 |

---

## 六、完整流程闭环

```
市场/用户反馈
     ↓
产品撰写 PRD
     ↓
设计出稿 → 评审
     ↓
开发实现 → 提测
     ↓
测试验证 → 产品验收
     ↓
上线部署 → 线上验证
     ↓
数据分析 → 复盘
     ↓
新一轮需求迭代
```

> 产品的加入让团队从「执行型」升级为「**产品驱动型**」，形成完整的需求 → 交付 → 反馈闭环。
