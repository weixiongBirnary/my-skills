# Project Anchor Map

## 概述

产出一份以代码证据为优先依据的项目高价值执行锚点地图。最终产物必须是 Markdown，不要输出 JSON、表格 DSL 或自定义标签格式。应从具体制品出发，例如路由、消息处理器、定时任务、CLI 命令、符号、依赖图和文件引用。

在撰写最终结果前，先阅读 [references/output-contract.md](references/output-contract.md)。
如果用户想看中文示例、格式样例，或希望在分析前先参考输出结构，请阅读 [references/example-cn.md](references/example-cn.md)。

## 优先使用的输入

如果提供了 `symbols`、`deps` 和 `entrypoints` 数据，优先使用它们。把这些制品视为线索；只要源文件可用，就要用文件路径和符号来核验关键结论。

使用以下锚点类型：

- 路由或 API 入口
- 消息或事件消费者
- 定时任务或后台任务
- CLI 命令或管理任务

如果可用证据中不存在用户要求的锚点类型，要明确说明，不要自行推断。

## 工作流

### 1. 盘点入口锚点

先列出所有可发现的入口点。对于每个候选锚点，记录：

- 触发类型
- 触发名称
- 所属文件
- 第一个具体符号

丢弃无法关联到文件路径和符号的候选项。

### 2. 重建核心调用路径

对于每个高价值锚点，重建从入口点到业务处理器再到基础设施边界的有序路径。优先给出 3 到 7 步，真实反映执行链路。

只使用能够从代码或给定图中证明的调用、注册、分发或导入关系。如果路径中断，就停在最后一个已证实的步骤，并标出缺口。

### 3. 提取实现锚点

找出最适合新工程师开始阅读该流程的位置。通常是以下之一：

- handler 函数
- 应用服务方法
- 编排类
- 命令执行器
- 工作流文件

要写出具体的函数、类或文件名，不要只做抽象概括。

### 4. 附上依赖和风险

只列出该路径实际触达的依赖，例如：

- 数据库或缓存
- MQ、流系统或调度器
- HTTP 客户端或第三方 SDK
- 对象存储或搜索系统

然后评估与该路径相关的运行风险：

- 鉴权或权限检查
- 事务边界
- 幂等性保障
- 重试策略
- 补偿或回滚行为

只有在存在明确证据，或“缺少证据本身就是风险”时，才可以声称存在风险。后一种情况请表述为 `No explicit ... evidence found in inspected path`。

### 5. 按业务价值排序

最终的前 10 个锚点要按可能的业务价值排序，而不只是按技术中心性排序。

优先使用以下排序启发式：

1. 营收、订单、支付或核心用户交易流程
2. 面向用户的主要产品流程
3. 保持业务运转所需的摄取或履约流程
4. 管理和支持流程
5. 内部工具或一次性实用脚本

如果证据不足以区分两个锚点，就明确说明排序是暂定的，并解释使用了哪些证据。

### 6. 识别“核心链路”视角

除了“业务价值最高”视角，还要支持“项目核心链路”视角。这里的“核心”不是泛指重要，而是优先寻找对系统运行具有中枢作用的入口流程，例如：

- 覆盖最多下游能力的主入口
- 汇聚多个子系统的编排入口
- 驱动状态流转或工件流转的主流程
- 被多个次级流程复用或依赖的上游入口

当用户说“最核心”“核心链路”“核心入口流程”时，优先按这个视角选取和排序；当用户明确说“业务价值最高”时，优先按业务价值排序。

### 7. 解析数量要求

从用户请求中解析输出数量约束。如果用户明确指定数量，按用户要求输出；如果没有明确指定，再使用默认值。

按以下规则处理：

- `全部`、`所有`、`完整`：输出全部有证据支撑的入口流程
- `一个`、`1 个`、`唯一最重要`：只输出 1 条最符合用户排序意图的入口流程
- `十个`、`10 个`：输出前 10 条
- `最核心`、`最关键`：默认输出 1 条；如果上下文显示用户想看一组核心链路，可输出 3 条并说明是核心候选
- 未指定数量：默认输出前 10 条

如果仓库内可证实的候选入口少于用户要求数量，就输出全部已证实项，并明确说明“可证实入口少于请求数量”。

### 8. 解析分组要求

当用户要求“按模块”“按域”“按子系统”“分别输出”时，不要只给一个全局列表，而要先把入口流程分组，再在每组内排序。

优先使用以下分组线索：

- entrypoint 所在目录
- handler 或 service 所在目录
- 明确的 bounded context、业务域或模块名
- schema、package、app、feature 目录边界

如果分组边界只有弱证据，就使用最保守的文件系统边界，并明确说明分组依据，例如 `按 src/payments、src/orders 目录分组`。

### 9. 解析“双视角对照”要求

当用户要求“业务视角和技术视角对照”“业务价值 vs 核心链路”“两种视角都给出”时，输出两个并列章节：

- 业务价值视角
- 核心链路视角

如果两种视角选出的入口流程高度重合，可以复用同一批锚点，但仍要明确说明排序依据不同。
如果两种视角结果明显不同，要简短指出差异原因，例如“一个更偏用户交易价值，另一个更偏系统编排中枢”。

## 输出规则

- 不要输出任何没有证据支撑的锚点。
- 不要虚构缺失的调用步骤、依赖或风险控制。
- 在 evidence 字段中使用精确的文件路径和符号名。
- 每个锚点都要紧凑、便于扫读。
- 最终结果必须是 Markdown 文件并放在当前项目的 `project-anchor-map/` 目录下。
- 如果用户要求模块化或双视角输出，章节结构要跟随用户意图调整，但仍保持 Markdown 可读性。
- 结尾必须包含 Mermaid 总览图和 `新同学阅读顺序` 章节。
- 用中文输出

## 证据缺失时的处理

当证据不完整时，使用以下模式之一：

- `Evidence gap: registration found, downstream handler not resolved from available artifacts.`
- `No explicit transaction boundary evidence found in inspected path.`
- `Dependency inferred from import plus callsite, but concrete runtime target is unresolved.`

把不确定性直接写在对应内容里，不要藏在免责声明中。
